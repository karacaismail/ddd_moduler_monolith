Framework Master Dökümanı

AI-first DDD Modüler Monolith — Tam Mimari

Frappe/Odoo/Drupal/Shopify/SAP esinli, 15 SaaS ürünü taşıyabilecek backbone. core-domains + eksikler birleşimi; her domain için derin dalış, gerçek dünya örnekleri, kod kontratları, junior tuzakları ve 15 ürün mapping tablosu.

00Yönetici özeti

Bu döküman, daha önce paylaştığın iki HTML'in (core-domains, eksikler) birleştirilmiş ve 20 kat derinleştirilmiş hali. Tahtanın mevcut hali tek bir veri-modelleme katmanı gibi düşünülmüş; oysa 15 üründen Shopify, SAP, Odoo, Frappe sınıfı bir şey çıkacaksa 6 katmanlı bir mimari gerek: Kum Tanesi (atomik tipler) → Küçük Taş (kernel) → Scale primitives (kernel-yakını) → Orta Taş (in-tree core) → Büyük Taş (Pages) → Services (dış).

Sentez bir cümlede: kernel'i mümkün olduğunca dar tut (6 domain), scale primitive'lerini save pipeline'a göm, in-tree core'ları Medusa "Infrastructure Module" disiplini ile swappable yap, business logic'i first-party stack module'lerine bırak, asıl ürünleri Layer 3 community marketplace'e at. Ölçek-uyumu kernel'e yerleştir ama policy değil, primitive olarak — feed fan-out gibi politika kararları her zaman module katmanında.

01Felsefe ve ana prensipler

1.1 Modüler monolit, mikroservis değil

Shopify'ın "majestic monolith"i 2.8M+ satırlık tek Rails uygulaması; Black Friday'de saniyede 80K request'i tek deploy unit ile karşıladı. Bizim hedefimiz aynı: tek deploy unit, çoklu bounded context, sıkı modül sınırı. Mikroservise sadece zorlandığında (extraction trigger) geç.

Modüler monolit'in mikroservise olan dört üstünlüğü, junior'a açıkla:

  • Refactor ucuz. Bounded context sınırını yanlış çizdin? Tek PR'da değiştir. Mikroservis olsa Kafka schema migration + 2 servis deploy + downstream konsumer'lar.
  • Local development tek komut. uvicorn app:app ile her şey çalışır. Mikroservis olsa Docker Compose'da 12 servis ayağa kaldır.
  • Distributed transaction yok. İki aggregate aynı transaction'da değiştirilebilir (gerçi DDD aggregate başına bir transaction kuralı vardır, ama in-process işin saga'sı çok daha basit).
  • Observability ucuz. Tek log stream, tek trace, tek metric source. Mikroservis dağıtık trace zorunlu.

Şartı: modül sınırı yazılım disipliniyle korunmalı. Shopify Packwerk ile yapıyor (Ruby static analysis cross-component call'ları flag'liyor). Sen Python'da importlinter + custom plugin ile aynı disiplini sağla — CI'da modül A modül B'nin internal'ına dokunursa fail.

1.2 Metadata-driven schema

Frappe DocType, Salesforce Object, Strapi Content-Type — üçü de aynı fikrin farklı isimleri: şema kodda değil, metadatada. Bir Customer tipi tanımlamak için Python class yazmak yerine, sistem tablosuna satır eklersin. Framework gerisini otomatik:

  • PostgreSQL tablosu otomatik oluşur (gerçek fiziksel tablo, JSON column değil — JOIN performansı için kritik).
  • REST + GraphQL endpoint'leri otomatik üretilir.
  • Admin form + list view otomatik render.
  • Permission engine satırları otomatik tanır.
  • Search index otomatik beslenir.
  • Audit log otomatik dinler.
  • Hook noktaları (before_save, after_insert) otomatik vardır.

Bu sayede müşteri "şu alanı eklemek istiyorum" dediğinde geliştirici ALTER TABLE yazmaz, migration deploy etmez — admin paneli açar, alanı ekler, hazırdır.

1.3 AI-first ne demek, ne demek değil

"AI-first" sloganı sürekli kötü kullanılıyor. Net tanım:

AI-first budur
  • Vector store kernel-adjacent primitive (pgvector default, Qdrant/Pinecone driver).
  • Her DocType save'inde otomatik embedding (opt-in, projection olarak).
  • Hybrid search (BM25 + vector RRF) Search domain'inde native.
  • MCP server her bounded context'te otomatik expose (her modül AI'a kendi tool'larını sunar).
  • Otomatik kategorize / anomali tespiti / tahmin event bus üstüne projection.
  • Agent orchestration kernel-adjacent (Pydantic AI tarzı type-safe).
AI-first bu değildir
  • "Bir yere ChatGPT integration ekleyip slogan basmak."
  • "Müşteri destek bot'u eklemek" (bu bir feature, mimari değil).
  • Her ürüne aynı LLM prompt'u dayatmak.
  • AI'ı bir module sanmak (gerçekte: AI primitives kernel-adjacent, AI applications module).

1.4 Polyglot hook — gerçek senaryo

Python core, TS opsiyon. Sebep: yazılım sektörünün önemli bir kısmı TS yazıyor; modülünü TS yazmak isteyen geliştiriciyi kaybetmemen lazım. Üç runtime stratejisi (her birinin yeri farklı):

  • Embedded JS (shared hosting, 5€/ay VPS): PythonMonkey (SpiderMonkey, aktif geliştirilmeye devam ediyor) veya STPyV8 (Cloudflare fork). Network round-trip yok. Trusted hook için.
  • Sidecar Node/Deno (production default): Node sürecine gRPC ile bağlan. Dapr modeli. İzole, bağımsız scale, debuggable.
  • WASM/Extism (untrusted marketplace plugin): Capability sandbox. "Bu plugin sadece şu DocType'ı okuyabilir, network erişimi yok" diye kısıtla.

Geliştirici tarafı tek interface görür: onBeforeSave, onAfterInsert, validate. Hangi runtime'a bağlandığını bilmez.

1.5 Scale primitives kernel'de, scale policy module'de

Bu döküman'ın en önemli prensiplerinden biri. Junior'a açıkla:

  • Primitive (kernel): Event bus, outbox, hot counter, projection. Hepsi nasıl ölçeklenirim sorusunun cevabını taşır.
  • Policy (module): Sosyal medya feed'inin push/pull/hybrid kararı, celebrity threshold (~10K follower), reservation expiration timing. Hepsi iş kuralı.

Sosyal medya turunda konuştuğumuz fan-out tartışması bu prensibin somutu. Twitter'ın timeline'ı kernel'e gömülmez; bunun yerine kernel fan-out worker primitivei sağlar, Feed module timeline politikasını yazar. 13 üründe feed yok; kernel onları taşımaz.

1.6 Türkiye uyumu birinci sınıf

Hetzner'de host edeceksin (Linux Debian, AMD EPYC). Müşterilerin TR'de. Yani e-Fatura, e-Defter, SGK, KVKK add-on plugin değil, kernel-adjacent. Aksi halde her ürünün muhasebe modülü tekerleği yeniden icat eder, her birinde aynı GİB entegrasyon kodu farklı bug'larla yaşar.

Yaklaşım: e-Belge Provider ve Tax Authority Adapter Layer 1 in-tree olarak otur. Driver pattern — Foriba/Sovos, İzibiz, Uyumsoft, KolayBi', Logo plug edilebilir. Default driver bir mock (development). Production'da geliştirici credentials girer, gerisi aynı.

03Layer 0 — Kernel (6 immovable domain)

Bu altısı kernel. Çıkarırsan framework yok. Gün-1'de bitir, sonradan retrofit'leme.

Layer 0 — Kernel #1

Schema / Metadata Engine (DocType)

Bu framework'ün kalbi. Frappe'nin DocType'ı, Odoo'nun Model'i, Salesforce'un Object'i, Strapi'nin Content-Type'ı — hepsi aynı fikrin farklı isimleri.

Domain modeli

Metadata kaydı şu beş tablodan oluşur (kavramsal, Frappe pattern):

doctype                # DocType definition (name, label, naming, permissions, options)
doctype_field          # Per-field metadata (fieldname, fieldtype, options, required, in_list_view, fetch_from)
doctype_permission     # Role-based default permissions (role, read, write, create, delete, submit, cancel, ifowner)
custom_field           # Tenant-specific extra field (extends doctype_field at runtime)
property_setter        # Tenant-specific override of any field property (e.g., make label different)

Bir Customer tipi tanımladığında framework şunları otomatik üretir:

  1. PostgreSQL'de tabCustomer tablosu — fiziksel tablo, alanlar fiziksel kolon (JSON polymorphism değil, JOIN performansı için kritik).
  2. GET /api/resource/Customer, POST /api/resource/Customer, PUT /api/resource/Customer/{id}, DELETE /api/resource/Customer/{id} REST endpoint'leri.
  3. GraphQL schema'sında type Customer + query Customer(id) + customers(filter) + mutation'lar.
  4. Admin paneli — list view, form view, kanban view, calendar view, gantt view (DocType metadata'da hangi view destekli olduğu yazılı).
  5. Permission engine — doctype_permission satırlarını okur, query layer'a otomatik WHERE injektsiyonu yapar.
  6. Search indexer — yeni satır eklendiğinde event publish eder, Search projection güncellenir.
  7. Audit logger — alan değişikliklerini tabVersion'a yazar (diff).
  8. Hook noktaları — before_insert, validate, before_save, after_save, on_submit, on_cancel, on_trash, after_delete, on_change.

Naming series — Frappe pattern

Her DocType'ın "autoname" stratejisi olmalı. Seçenekler:

Submittable DocType

Bazı kayıtlar "draft → submitted → cancelled" mantığında yaşar (faturalar, siparişler, journal entry). Frappe'de is_submittable: 1 diyince DocType'a docstatus alanı eklenir (0=draft, 1=submitted, 2=cancelled). Submit edildikten sonra belli alanlar değiştirilemez (financial integrity). Sadece on_update_after_submit hook'unda belirli alanlar açıkça izinli.

Child Table (Table fieldtype)

Order → OrderLine ilişkisi. Frappe pattern: child DocType, istable: 1 ile işaretli, parent'a parent, parenttype, parentfield alanlarıyla bağlı. Lifecycle parent'a bağlı (parent silinince child silinir).

Niye CORE

Identity de DocType (User), Authz de DocType (Role, Permission), Tenancy de DocType (Site/Tenant), Workflow da DocType (Workflow, WorkflowState). Hepsi kendini schema engine üstüne tanımlar. Çıkarırsan framework yok.

Tuzak — JSON column polymorphism

Junior çoğunlukla şöyle başlar: tek bir records tablosu, data JSONB kolonu, "her şey JSON". WordPress'in wp_postmeta pattern'i bu — felaket. Çünkü WHERE/ORDER BY/JOIN performansı çöker, index'leme zor, type safety yok. Frappe doğru yolu seçti: her DocType kendi fiziksel tablo. Sen de aynı yap; JSON sadece schema-less ek alan için.

Layer 0 — Kernel #2

Identity

Her request'in başında "kim bu" sorusunu cevaplayan ana mekanizma. Login/signup'tan çok daha fazlası.

Domain modeli

User                  # email, password_hash (Argon2id), status, locale, timezone
Session               # session_id, user_id, ip, user_agent, expires_at
Token                 # personal access token, OAuth token, refresh token
TwoFactorMethod       # TOTP secret, WebAuthn credential, SMS phone
SSOConnection         # OIDC provider, SAML IdP, OAuth app config
ImpersonationLog      # admin_user, target_user, started_at, ended_at, reason
PasswordReset         # token, user_id, expires_at, used_at
ServiceAccount        # machine identity, scoped credentials

Yetenekler

  • Şifre auth — Argon2id (RFC 9106), cost 19MiB/2 iter/1 lane.
  • Session — server-side Redis session (cookie ID), JWT opsiyonel (stateless API için).
  • OAuth/OIDC — Google, Microsoft Entra ID, Apple, GitHub, generic OIDC.
  • SAML 2.0 — enterprise SSO (Okta, Auth0, Azure AD).
  • 2FA — TOTP (RFC 6238, Google Authenticator), WebAuthn (FIDO2 passkey), backup codes, SMS fallback.
  • Magic link — passwordless email login.
  • Personal Access Token (PAT) — developer API key, scoped.
  • Service account — server-to-server auth, key rotation, scoped.
  • Impersonation — admin başka kullanıcı olarak giriş, audit log'a "X impersonated Y at HH:MM" yazar.
  • Password reset — token tek-kullanımlık, 30 dk TTL, brute force koruması.
  • Account recovery — backup email, social recovery (FIDO2 + recovery phrase).
  • Rate limit — failed login backoff (1 → 2 → 4 → 8 saniye), IP-bazlı, account-bazlı.
  • Device management — "şu cihazlar oturum açık" listesi, remote logout.
  • Anonymization (KVKK) — hesap silindi flag, email/name/phone null'lanıp tutarlılık için ID korunur.

Layer 0 — Kernel #3

Authorization

Authentication = "kim", Authorization = "ne yapabilir". Karıştırılmasın. Authz query layer'a örülmeli; application-level if-check yetmez.

Üç model birlikte

Gerçek hayatta üç modeli birlikte kullanırsın:

  • RBAC (Role-Based) — admin, editor, viewer rolleri. Klasik. Frappe'nin Role + DocPerm sistemi.
  • ABAC (Attribute-Based) — "kullanıcının departmanı IT ise saat 09-18 arası ise iç ağdaysa". Koşul-temelli. SpiceDB'de "caveats" olarak modellenir.
  • ReBAC (Relationship-Based) — "kullanıcı bu document'in owner'ı ise" veya "bu projenin member'ı ise". Google Zanzibar paper'ı, OpenFGA, SpiceDB temeli.

OpenFGA modeli — kanonik

Senin framework'ünde OpenFGA-style DSL ile authz tanımla:

model
  schema 1.1

type user

type organization
  relations
    define member: [user]
    define admin: [user]

type document
  relations
    define parent: [organization]
    define owner: [user]
    define editor: [user] or owner or admin from parent
    define viewer: [user] or editor or member from parent

Sonra check(user:alice, viewer, document:report-2026) diye sorgu yaparsın. Sistem tuple store'a bakar: document:report-2026.parent = organization:acme + organization:acme.member = user:alice → true.

Query layer injection — kritik

Frappe'nin permission_query_conditions hook'u — junior'a bunu illa öğret:

# hooks.py
permission_query_conditions = {
    "SalesOrder": "myapp.permissions.sales_order_conditions"
}

# myapp/permissions.py
def sales_order_conditions(user):
    if not user:
        user = framework.session.user
    if "Sales Manager" in get_roles(user):
        return ""  # No restriction
    # Sales rep sadece kendi siparişlerini görür

Bu fonksiyon her SELECT * FROM tabSalesOrder sorgusunda WHERE'e otomatik AND'lenir. Application kodu ona bakmaz. Module geliştirici yanlışlıkla db.query("SELECT...") yazarsa bile WHERE injected.

PostgreSQL Row-Level Security entegrasyonu

OpenFGA pratikte aplikasyon layer'ında çalışır. Tenant izolasyonu için defense-in-depth olarak PostgreSQL RLS de aç:

ALTER TABLE tabSalesOrder ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation ON tabSalesOrder
USING (tenant_id = current_setting('app.tenant_id')::uuid);

-- Session başında:
SET LOCAL app.tenant_id = 'acme-tenant-uuid';

Performans: tenant_id üzerinde index varsa overhead %2-5. PgBouncer transaction mode'da SET LOCAL kullan (session değişkeni tx scope'una bağlanır, cross-tenant leak imkansız).

Field-level security

Bazı alanlar belli rolde görünmez. HRMS senaryosu: çalışan listesinde maaş alanı sadece HR rolüne görünür. DocType metadata'da:

doctype_permission:
  - role: Employee, permlevel: 0, read: 1
  - role: HR, permlevel: 1, read: 1, write: 1

Resource sharing — Drive pattern

Drive (#7) ürünü için: kullanıcı dosyayı belli kişilerle paylaşır. ReBAC tuple ekler: document:photo-summer-2026.viewer = user:bob. Sonraki query'de Bob'un access listesi bu dosyayı içerir.

En kritik kural

Authz kesinlikle module olamaz. Query layer'a örülmesi zorunlu. Module olsa, bir geliştirici manuel SELECT * yazıp WHERE eklemeyi unutursa — bir tane unutsa tüm tenant'ların verisi sızar. Frappe'nin permission_query_conditions pattern'i, OpenFGA'nın "List Objects" API'si, PostgreSQL RLS — üçü birlikte defense-in-depth.

Layer 0 — Kernel #4

Tenancy

"SaaS yapıyoruz" = "aynı yazılımı çok müşteriye, herkesin verisi izole" demek. Tenancy bunun mekanizması.

Üç model

  • Single-tenant (database-per-tenant) — her müşteri kendi DB'si. Enterprise müşteri "verim kimseyle aynı yerde olmasın" diyorsa. Operasyon cehennemi, migration zorlaşır.
  • Multi-tenant shared schema + tenant_id — tek DB, tüm tablolarda tenant_id kolonu. En yaygın. Default'umuz bu.
  • Multi-tenant schema-per-tenant — tek DB, her tenant kendi PostgreSQL schema'sı. Backup/restore kolay, izolasyon güçlü; 10K+ tenant'ta migration zorlaşır.
  • Hybrid (Shopify pattern) — küçük tenant'lar shared "pod"larda, dev müşteriler dedicated pod. Pod-aware routing.

İki seviyeli tenancy — muhasebe senaryosu

Mali müşavir bürosu senin müşterin. Bürosunun 50 mükellefi var, her mükellefin defteri ayrı. İki seviyeli tenancy:

tenant_id   = office_uuid       # mali müşavir bürosu (faturalama, login)
company_id  = client_company    # mükellef şirket (defter, fatura, beyanname)

-- Her finansal kayıtta her iki kolon
CREATE TABLE journal_entry (
  id uuid PRIMARY KEY,
  tenant_id uuid NOT NULL,
  company_id uuid NOT NULL,
  ...
);
CREATE INDEX ON journal_entry (tenant_id, company_id);

RLS policy iki seviyeyi de kontrol eder. Bunu modellemezsen büro tüm mükellefleri karıştırır.

Tenant ≠ User

Bir user birden çok tenant'ta üye olabilir (Slack workspace switcher). User ↔ Tenant many-to-many:

CREATE TABLE membership (
  user_id uuid REFERENCES users(id),
  tenant_id uuid REFERENCES tenants(id),
  role text NOT NULL,
  joined_at timestamptz NOT NULL,
  PRIMARY KEY (user_id, tenant_id)
);

Tenant ≠ Site

Bir tenant birden çok site (subdomain, custom domain) işletebilir. CMS müşterisinin 5 marka sitesi olabilir, hepsi aynı tenant'a ait. site tablosu ayrı, tenant_id'ye bağlı.

Custom domain — e-ticaret SaaS

Shopify gibi: her merchant kendi domain'ini kullanır (acme-store.com). DNS provisioning (CNAME), SSL certificate (Let's Encrypt + ACME), routing (host header → tenant lookup). Senin "Domain" küme'n bunu modelliyor — DNS + SSL/TLS + Wildcard.

# Tenant lookup middleware
@app.middleware("http")
async def tenant_resolver(request, call_next):
    host = request.headers["host"]
    # 1. Custom domain mi?  acme-store.com → tenant lookup
    # 2. Subdomain mi?       acme.myplatform.com → "acme" prefix
    # 3. Path-based mi?      myplatform.com/acme/... → path prefix
    tenant = await resolve_tenant(host, request.url.path)
    request.state.tenant = tenant
    # PostgreSQL RLS context
    await db.execute(f"SET LOCAL app.tenant_id = '{tenant.id}'")
    return await call_next(request)
Tuzak #1 — "önce tek tenant"

"İlk müşterim için tutarlı çalışsın, sonradan çoklarım" düşüncesi. Kaybedilen 6-12 ay. Çünkü retrofit = tüm tabloları rebuild + tüm query'leri rewrite + cross-tenant leak penceresi. Day-1'de tenant_id her satırda.

Tuzak #2 — PgBouncer + RLS

PgBouncer transaction mode kullanıyorsan SET LOCAL app.tenant_id yapmalısın (transaction scope). SET (session scope) kullanırsan bir sonraki transaction farklı tenant'a bağlanır → veri sızıntısı.

Layer 0 — Kernel #5

Event Bus + Polyglot Runtime Bridge

Module'lerin loose-coupled konuştuğu omurga. Aynı kod, üç ölçek: in-process → Redis Streams → Kafka, module değişmeden.

Domain modeli — event nedir

Event {
  id: uuid                # idempotency key
  type: "OrderPlaced"     # event type
  aggregate_id: "order-123"
  aggregate_type: "Order"
  tenant_id: uuid
  occurred_at: timestamptz
  occurred_by: user_id
  version: 1
  payload: jsonb          # event data
  trace_id: text          # distributed tracing
}

İki ayrı kavram — karıştırma

  • Domain Event — "bir şey oldu, sistem state'i değişti": OrderPlaced, CustomerCreated, PaymentReceived. Past tense, immutable.
  • Command — "şu işi yap": PlaceOrder, CreateCustomer. Future tense, validated, can fail.

Event bus'ta sadece event akar. Command'ları doğrudan handler çağırır (genelde Aggregate.do_something method'u). Event Sourcing dünyasına gitmeden bunu net tut.

Üç transport, tek interface

from framework.events import publish, subscribe

# Publish
await publish(OrderPlaced(order_id="123", total=199.00))

# Subscribe
@subscribe(OrderPlaced)
async def decrement_stock(event: OrderPlaced):
    for item in event.items:
        await stock.decrement(item.sku, item.qty)

Backend deployment profile'a göre seçilir:

  • Shared hosting: InProcessBus — async loop, tek process.
  • Orta ölçek: RedisStreamsBus — persistence, consumer groups, replay.
  • Büyük ölçek: KafkaBus — partition, retention, replay, multi-consumer.

Module kodu publish/subscribe'dan başkasını bilmiyor. Config flag: EVENT_BUS=redis.

Üç runtime stratejisi

Hook contract aynı, runtime swappable:

  • Embedded JS (shared hosting) — PythonMonkey (SpiderMonkey, Distributive Inc., 2024-2026 aktif), STPyV8 (Cloudflare fork). Tek process, network yok, latency düşük. Trusted hook için.
  • Sidecar Node/Deno (production default) — Node sürecine gRPC ile bağlan, MCP-style. Bağımsız ölçeklenir, debuggable, isolation güçlü.
  • WASM/Extism (untrusted plugin) — capability sandbox. Marketplace plugin'i için. "Bu plugin sadece şu DocType'ı okuyabilir, network yok" diye kısıtla.

Aggregate boundary kuralı — DDD kritik

TS hook Python aggregate'i üzerinde açık transaction tutamaz. Distributed transaction (2PC) çağı geçti — yavaş, fragile, gerçekte kullanılmıyor. Doğru pattern'ler:

  • Synchronous hook = pure function. Python core transaction açar, TS hook'a aggregate kopyasını verir, hook decision/mutation döner, Python core commit eder. Frappe'nin doc_events pattern'i.
  • Cross-aggregate effect = outbox + saga. Hook farklı aggregate'i etkileyecekse, transaction'da olmaz. Outbox'a event yazılır, ayrı consumer asenkron işler, başarısızlıkta compensating action.

Event versioning

Event'ler kontrat. Versiyon değişimi backward-compat zorunlu:

  • Yeni alan eklemek = OK (optional).
  • Alan silmek = breaking. Yeni event tipi yarat (OrderPlacedV2).
  • Anlam değişikliği = breaking. Yeni tip.
  • Event store retention (replay için) varsa migration script gerekir.

Scale primitive #1

Transactional Outbox

Junior'a: Dual-write problem. Order'ı DB'ye yazıp Kafka'ya event publish etmek — ikisi iki transaction. Biri başarılı, biri başarısız olabilir.

Çözüm: aynı DB transaction'da Order satırı + Outbox satırı INSERT et. Sonra ayrı relay worker outbox'tan Kafka'ya publish eder.

CREATE TABLE outbox (
  id uuid PRIMARY KEY,
  aggregate_type text NOT NULL,
  aggregate_id text NOT NULL,
  event_type text NOT NULL,
  payload jsonb NOT NULL,
  tenant_id uuid NOT NULL,
  status text NOT NULL DEFAULT 'pending',  -- pending|published|failed
  attempts int NOT NULL DEFAULT 0,
  occurred_at timestamptz NOT NULL DEFAULT now(),
  published_at timestamptz,
  next_retry_at timestamptz
);
CREATE INDEX outbox_pending_idx ON outbox (status, next_retry_at) WHERE status = 'pending';

Relay worker (Python + asyncpg):

async def outbox_relay():
    while True:
        async with db.transaction():
            rows = await db.fetch("""
                SELECT * FROM outbox
                WHERE status = 'pending' AND (next_retry_at IS NULL OR next_retry_at <= now())
                ORDER BY occurred_at
                LIMIT 100
                FOR UPDATE SKIP LOCKED
            """)
            for row in rows:
                try:
                    await kafka.publish(row['event_type'], row['payload'])
                    await db.execute("UPDATE outbox SET status='published', published_at=now() WHERE id=$1", row['id'])
                except Exception as e:
                    backoff = min(60 * 2 ** row['attempts'], 3600)
                    await db.execute("""
                        UPDATE outbox SET attempts = attempts + 1,
                          next_retry_at = now() + interval '1 second' * $1
                        WHERE id = $2
                    """, backoff, row['id'])
        await asyncio.sleep(1)

FOR UPDATE SKIP LOCKED concurrent worker için kritik. Birden çok worker aynı outbox'tan paralel okuyabilir, lock contention yok.

Debezium alternatifi: PostgreSQL logical replication üstüne Debezium outbox tablosunu Kafka'ya stream eder. Python tarafında sadece INSERT yeterli. Production'da tercih edilen pattern.

Retention

Outbox tablosu büyür. 2-billion satıra ulaşan fintech örneği var. Politika:

  • published 7 gün sonra archived table'a taşı.
  • failed 30 gün sonra dead letter queue + alert.
  • Partitioning: monthly partition (PostgreSQL declarative partition).
Scale primitive #2

CQRS Projections (Read Models)

Write model normalize, transaction-safe; read model denormalize, hız için. Event-driven olarak senkronize.

DocType'a projection metadata

doctype: CustomerStats
is_projection: true
source_events: [OrderPlaced, OrderCancelled, OrderRefunded]
rebuild_strategy: replay  # event store'dan baştan kur

projection_handler:
  OrderPlaced: |
    customer = stats.get(event.customer_id)
    customer.total_spent += event.total
    customer.order_count += 1
    customer.last_order_at = event.occurred_at
  OrderCancelled: |
    customer = stats.get(event.customer_id)
    customer.total_spent -= event.refund_amount

Framework bunu görür, otomatik subscriber kurar. Projection rebuild komutu: framework rebuild-projection CustomerStats — event store'dan baştan replay eder (durable workflow ile, kesintide kaldığı yerden).

Eventual consistency penceresi

Event işlenene kadar 1-5 saniye gecikme tipik. Dashboard "müşterinin total spent"i 2 saniye stale olabilir. Çoğu zaman OK. Ödeme/finansal sayfalarda fresh-read gerekirse: ?fresh=true param → write model'den oku (yavaş ama kesin).

Scale primitive #3

Hot Counter

Like sayacı, view sayacı, follower sayacı, stok-on-hand. Milyonlarca eşzamanlı update.

API

counter.increment("post:123:likes", by=1)
counter.read("post:123:likes")  # approximate, eventual

Backend strateji deployment'a göre

  • Shared hosting: UPDATE posts SET likes = likes + 1 WHERE id = ?. Yeterli.
  • Orta ölçek: Redis INCR, periodic flush DB'ye (10 saniyede 1).
  • Büyük ölçek: sharded counter — likes:post:123:shard_0, shard_1...shard_15. Read'de SUM. Approximate, kesinlik feda.

API tek interface. Module geliştirici backend'i bilmez. Twitter/X bu pattern'i kullanır.

Scale primitive #4

Workers + Scheduler

Async job queue. Outbox relay, projection rebuild, fan-out, scheduled task, webhook delivery — hepsi burada.

Job kategorileri

  • Immediate — queue'ya at, en yakın worker alır. (örn: email send)
  • Scheduled — 15 dk sonra çalışsın. (örn: kart abandonment reminder)
  • Recurring — her gün 02:00. (örn: daily cleanup)
  • Priority — yüksek (payment webhook), normal, düşük (marketing email).

  • Shared hosting: arq (asyncio Redis queue) tek worker process.

Scale primitive #5

Saga + Compensation

Cross-aggregate transaction. "Order place → Payment charge → Stock reserve → Shipping create" — herhangi biri fail ederse önceki adımları geri al.

Orchestration vs Choreography

  • Orchestration: merkezi koordinatör adım sırasını ve kompansasyonu bilir. Debug kolay, test edilebilir. Modüler monolit için tercih.
  • Choreography: servisler event'lere reaktif. Merkezi koordinatör yok. Mikroservis için.

DBOS implementasyonu

from dbos import DBOS, WorkflowID

@DBOS.workflow()
async def place_order_saga(order_id: str):
    try:
        payment = await DBOS.transaction(charge_payment)(order_id)
        stock = await DBOS.transaction(reserve_stock)(order_id)
        shipping = await DBOS.transaction(create_shipment)(order_id)
        await DBOS.transaction(finalize_order)(order_id)
        return order_id
    except StockUnavailable:
        await DBOS.transaction(refund_payment)(payment)
        raise
    except ShippingFailed:
        await DBOS.transaction(release_stock)(stock)
        await DBOS.transaction(refund_payment)(payment)

Hybrid search SQL pattern

WITH semantic AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY embedding <=> $1) AS rank
  FROM docs
  WHERE tenant_id = current_setting('app.tenant_id')::uuid
  ORDER BY embedding <=> $1 LIMIT 50
),
keyword AS (
  SELECT id, ROW_NUMBER() OVER (ORDER BY ts_rank_cd(tsv, query) DESC) AS rank
  FROM docs, plainto_tsquery('turkish', $2) query
  WHERE tenant_id = current_setting('app.tenant_id')::uuid
    AND tsv @@ query LIMIT 50
)
SELECT id,
  COALESCE(1.0/(60+s.rank), 0) + COALESCE(1.0/(60+k.rank), 0) AS score
FROM semantic s FULL OUTER JOIN keyword k USING (id)
ORDER BY score DESC LIMIT 20;

7.2 Security Model

  • Threat model: STRIDE — Spoofing/Tampering/Repudiation/Information disclosure/DoS/Elevation.
  • Defense in depth: app-level authz + DB RLS + WAF + rate limit.
  • OWASP Top 10: SQLi (parameterized), XSS (auto-escape), CSRF (token), SSRF (URL allowlist), open redirect, broken auth (yukarıda).
  • Secret rotation: Vault driver, otomatik rotate.
  • PII encryption: field-level, AES-256-GCM, key versioning.
  • Audit log immutability: append-only, ihlal forensics için.
  • Dependency scanning: dependabot, snyk, trivy.
  • Pentest: yıllık external.
  • Bug bounty: HackerOne, BugCrowd opsiyonel.

7.3 Observability

  • Logging: structlog (Python), JSON format, trace_id correlation, Loki/ELK.
  • Metrics: Prometheus + Grafana. RED method (Rate, Errors, Duration).
  • Tracing: OpenTelemetry, Jaeger/Tempo. Per-request trace, multi-tenant tagging.
  • Error tracking: Sentry. Source map, breadcrumb.
  • Real User Monitoring (RUM): Datadog RUM, Sentry Performance.
  • Alerting: PagerDuty / Opsgenie / Slack.
  • Status page: statuspage.io / Cachet self-host.

7.4 Performance Budget

  • API p50 < 50ms, p95 < 200ms, p99 < 500ms.
  • Page render TTFB < 200ms, LCP < 2.5s.
  • DB query budget per endpoint: max 10 queries (N+1 prevention).
  • Cache hit rate > 80%.
  • Background job throughput per worker > 100 job/s.

7.5 Backup & DR

  • PostgreSQL: pg_basebackup + WAL archiving (Barman, pgBackRest).
  • RPO < 5 dakika, RTO < 1 saat.
  • Cross-region replica (Hetzner FSN1 → HEL1).
  • Backup encryption (at-rest).
  • Restore drill: ayda 1 test.
  • Tenant-aware backup: UpdraftPlus pattern, müşteri kendi backup'ını alabilir.
