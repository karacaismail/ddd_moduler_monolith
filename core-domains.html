00Mimari kademe

Framework dört katmana ayrılır. Bir domain'in hangi katmanda yaşadığı tek bir testle belirlenir: çıkarınca framework çalışmaya devam eder mi? Eden katman, etmeyen katman.

Layer 0
Kernel
Çıkarırsan framework ölür. Beş immovable domain. Frappe'nin frappe paketi, Salesforce'un multitenant kernel'i.
Layer 1
In-tree core
Default kurulumla gelir, ama interface arkasındadır. Medusa "Infrastructure Module" pattern'i. Topluluk değiştirebilir.
Layer 2
First-party module
Sen yazarsın, isteyen kurar. Billing, Pricing, Reporting, AI, Location, Feed gibi kategori-spesifik şeyler.
Layer 3
Community / app
Üçüncü taraf marketplace. 15 ürünün business logic'i (Listing, BOM, Stock Ledger, Social Graph...) buraya iner.
Kuralın özü

Frappe core ile ERPNext'in ilişkisini örnek al. frappe paketinde DocType, User, Workflow, Notification, File, Audit, Search, Communication, Calendar, Scheduler, multi-tenancy var. ERPNext'te ise Customer, Supplier, Invoice, BOM var. Generic olan core'da, business semantic taşıyan modülde. Sen de aynı çizgiyi çek.

01Layer 0 — Kernel (5 değişmez)

Bu beşi çıkarırsan kalan her şey çöker. Bu beşi sonradan retrofit'lemeye çalışırsan 6 ay kaybedersin. O yüzden gün-1'de bitirip kapat.

Layer 0 — Kernel

Schema / Metadata Engine (DocType)

Junior'a: Normal Django/Rails projesinde "Customer" tablosu yapmak için CREATE TABLE yazıyorsun. Yeni alan eklemek için ALTER TABLE + migration. Bu framework'te öyle çalışmıyor. Sen sadece bir metadata kaydı oluşturuyorsun: "Customer adında bir DocType istiyorum; name, email, phone, tier alanları olsun". Framework gerisini otomatik yapar.

Tek bir DocType tanımından framework şunları otomatik üretir:

  • DB tablosu — Postgres'te tablo + index'ler. (Salesforce gibi gerçek implementasyonlarda fiziksel tablo bile yok, sparse columnar storage; sen Postgres ile başla, çalışır.)
  • REST + GraphQL endpoint'leri — GET /api/customer, POST /api/customer, list/filter/sort dahil.
  • Admin form'u — alan tiplerine göre input render eder (text, select, date, link-to-other-DocType, table-in-table).
  • List view'u — sortable, filterable, kolay özelleştirilebilir.
  • İzin satırı — Authorization sistemine otomatik kayıtlı (kim okur, kim yazar).
  • Search index'i — yeni satır eklendiğinde otomatik index'e gider.
  • Audit log entry — kim hangi alanı ne zaman değiştirdi, otomatik kayıt.
  • Hook noktaları — before_insert, after_save, on_submit gibi lifecycle event'leri otomatik var.
Gerçek dünya örnekleri
  • E-ticaret (Shopify-tarzı): "Product" DocType. Mağaza sahibi panelden "Bedeni" diye yeni bir alan ekliyor. ALTER TABLE yok, deploy yok, downtime yok. Metadata tablosuna 1 satır, sonraki request'te alan görünür.
  • Restaurant POS: "MenuItem" DocType. Bir restoran "allergens" alanı (multi-select), başka restoran "spice level" (1-5 slider) istiyor. Aynı kod, farklı tenant, farklı şema. Çünkü şema metadata olarak tenant'a bağlı.
  • CRM (Pipedrive-tarzı): "Deal" DocType. Sales manager bugün "campaign source" alanını eklemek istiyor. Geliştiriciye soruyor mu? Hayır, kendi ekliyor. Çünkü DocType editor admin panelinde.
  • Real estate (#1): "Listing" DocType. Konut tipinde "balkon var mı" boolean alanı; tarla tipinde "imar durumu" select alanı. Aynı DocType, conditional fields.
  • LMS (#6): "Course" DocType. Kurum A "prerequisite" çoklu link alanı ister; Kurum B "instructor signature" image alanı. İkisi aynı framework, ikisi farklı tenant.
  • HRMS (#9): "Employee" DocType. Türk firma için "TC kimlik", "SGK numarası". Alman firma için "Steueridentifikationsnummer". Tek DocType template'i, locale-aware extension.

Niye CORE (modüle inmez)

Hiçbir domain bunsuz tanımlanamaz. Identity bile bir DocType (User). Authz bile DocType (Role, Permission). Tenancy bile DocType (Site/Tenant). Hepsi kendini şema motoruyla tanımlar. Çıkarırsan framework yok.

Tuzak — yarım shipping

Drupal 7 entity API'yi yarım shipping etti. Topluluk eksiği kapatmak için contrib Entity modülü yazdı. D8'e taşırken herkes manuel migration yaptı, 5 yıl acı. Schema engine'i tam bitir. Eksik shipping = 5 yıllık ekosistem kanıtması.

Tuzak — "WordPress benzeri çözerim"

WordPress'in wp_postmeta tablosu polymorphic key-value. Filtreleme yavaş, JOIN cehennemi. Frappe/Salesforce öyle yapmıyor — her DocType kendi tablosu (Frappe) veya sparse columnar (Salesforce). Sen Postgres ile her DocType'a fiziksel tablo aç, JSON kolonu sadece schema-less ek alanlar için kullan.

Layer 0 — Kernel

Identity

Junior'a: "Login/signup'ın olduğu yer" diye düşünüyorsan eksik. Identity, framework'ün geri kalanına "şu an gelen request'i kim yaptı" sorusunu cevaplayan ana mekanizma. Her endpoint çağrısında, her DocType save'inde, her event publish'te sistemin "current_user" diye bildiği şey buradan gelir.

İçinde olması gereken her şey:

  • User entity — email, password hash (Argon2id), profile data, status (active/suspended/deleted).
  • Session management — cookie session, JWT, refresh token rotation, session revocation.
  • OAuth / SSO — Google, GitHub, Microsoft Entra ID, generic OIDC. SAML 2.0 enterprise için.
  • 2FA — TOTP (Google Authenticator), SMS (fallback), passkey/WebAuthn (modern standart).
  • Personal access token / API key — geliştirici curl ile API test etsin diye.
  • Service account — server-to-server, user'a bağlı olmayan auth.
  • Impersonation — destek ekibi "müşteri olarak giriş yap" yapsın, audit trail'e yansısın.
  • Password reset — token expire'lı, one-time, brute force koruması.
  • Account recovery — backup code, alternative email, social recovery.
  • Rate limit + brute force koruması — failed login attempt sayacı, exponential backoff, IP/account level.
Gerçek dünya örnekleri
  • B2B SaaS (Linear, Notion): şirket admin'i çalışanlarını sadece Google Workspace ile login etsin istiyor. Şifre kullanmasınlar. SSO core'da olduğu için config'den açıyorsun.
  • Bankacılık app: kullanıcı her giriş için passkey (yüz tanıma + cihaz). WebAuthn core'da olunca custom yazmıyorsun, hazır.
  • Geliştirici dokümanı: developer API'sini Postman'de test etmek için pat_abc123 üretiyor. Sebep: kendi şifresini Postman'e yazmak istemiyor.
  • Müşteri destek senaryosu: destek temsilcisi "kullanıcı görmüyor" diyor — admin onun hesabına geçici giriş yapar (impersonate). Sistem log'a "X kullanıcısı, Y kullanıcısının hesabına 14:32-14:38 arası giriş yaptı" diye yazar.
  • Sosyal medya (#14) ölçeği: 500M kullanıcı, login akışı saniyede 100K request. Session token Redis'te, rate limit Redis'te. Core'da bu primitive'ler hazır.
  • Compliance senaryo: KVKK / GDPR'a göre kullanıcı hesabını sildiğinde data anonymize edilmeli. Identity core'da "soft delete + anonymization pipeline" hazır gelir.

Niye CORE

Her request'in başında bu soru var. Module olamaz çünkü diğer module'ler authentication olmadan ne hook çağırabilir, ne event işleyebilir.

Junior tuzağı — "Auth'umu kendim yazarım"

Demek istediği şey: bcrypt cost ayarlamak, timing-safe karşılaştırma yapmak, password reset token'ı tek-kullanımlık ve süre dolumlu tutmak, session fixation'a karşı korunmak, OAuth state parameter doğrulamak, JWT secret rotation yapmak. Junior'ın 3 hafta yazıp bug bırakacağı şey. Core'a koymanın sebebi: her ürün ihtiyaç duyar, yanlış yazılınca felaket olur.

Layer 0 — Kernel

Authorization

Junior'a: Authentication ile karıştırılmasın. Authentication = "sen kimsin, kanıtla" (login). Authorization = "sen busun, ne yapabilirsin". İkisi farklı domain. Bir kullanıcı login olabilir (auth pass) ama o sayfaya erişim izni olmayabilir (authz fail).

Modeller (junior'ın bilmesi gereken 3 model):

  • RBAC (Role-Based): "admin", "editor", "viewer" rolleri. Her rolün hak setı var. Klasik.
  • ABAC (Attribute-Based): "kullanıcının departmanı IT ise + saat 09-18 arası ise + IP iç ağdan ise". Koşul-temelli.
  • ReBAC (Relationship-Based): "kullanıcı bu dokümanın owner'ı ise" veya "bu projenin member'ı ise". Google Zanzibar paper'ı, OpenFGA, SpiceDB bu model üzerine.

Senin framework'ünde üçü de gerekli. Authz layer'ı query layer'a örülmeli (query'ye otomatik WHERE eklenmeli) ki module geliştirici yanlışlıkla bile data leak yapamasın.

Frappe'nin gerçek tekniği

Frappe'de permission_query_conditions hook'u var. Sen "Order DocType'a herkes sadece kendi siparişini görsün" derken aslında her SELECT * FROM tabOrder query'sine framework otomatik WHERE owner = current_user OR ... ekler. Module geliştirici hiç düşünmüyor. Authz query layer'ında, application layer'ında değil.

Gerçek dünya örnekleri
  • Trello: board'un member'ları sadece o board'u görür. Karta atanmış kişi karta erişebilir. ReBAC — "X, board:Y'nin member'ı" relation tuple'ı.
  • Gmail: bir mail'in receiver'ısın → okuyabilirsin. Sender'sın → "Sent" klasöründe silebilirsin ama receiver'da silinmez. ABAC — attribute (sender/receiver) + action (read/delete) eşlemesi.
  • E-ticaret admin paneli: "customer-service" rolü sipariş görür, sadece "refund" yetkisi var ama 1000 TL üstü onaylayamaz. Manager 10K TL altını onaylar, üstünü CFO. RBAC + ABAC karışımı.
  • Drive/Dropbox (#7): dosya paylaşımı — link share, kişi share, klasör share, organization-wide share. Hepsi resource-başına ACL. Çoklu kombinasyon ReBAC.
  • GitHub: repo'nun maintainer'ı issue kapatabilir, contributor PR açabilir, public viewer sadece okur. Org içinde takım izinleri repo'lara cascade eder. Hierarchical ReBAC.
  • HRMS (#9): İK uzmanı tüm maaşları görür; yönetici sadece direct report'ların maaşını görür; çalışan sadece kendininkini. Field-level + relationship-based.
  • Healthcare: doktor sadece kendi hastasının kaydını görür. Acil servis doktoru "break-glass" ile başkasınınkine erişir ama audit log'a kırmızı flag düşer.

Niye CORE — kritik teknik nokta

Authz module olamaz. Çünkü query layer'a örülmesi gerekir. Eğer module olsaydı, her geliştirici manuel WHERE tenant_id = ? AND owner = ? yazmayı unutur — bir tane unutsa bir tenant'ın verisi diğerine sızar. Tenancy ile birlikte gün-1 kernel disiplini.

Tuzak — application-level check

Junior çoğunlukla şöyle yazar:

orders = db.query("SELECT * FROM orders")
return [o for o in orders if o.user_id == current_user.id]

Bu yanlış: DB'den tüm satırlar geldi, sonra filter. 10M satırlık tabloda bellek patlar. Doğrusu: query layer authz'i otomatik WHERE'e çevirir, DB'den sadece izinli satır gelir.

Hangi araç

OpenFGA (Auth0/Okta) veya SpiceDB (Authzed) inceleyebilirsin. İkisi de Zanzibar implementation'ı, gRPC API'leri var, kendi sidecar'larında çalışıyor. Sen framework'ün authz layer'ını OpenFGA-style relation tuple'lar üzerine kurabilirsin (in-process Python lib + opsiyonel sidecar).

Layer 0 — Kernel

Tenancy

Junior'a: "SaaS yapıyoruz" demek aslında "aynı yazılımı bir sürü müşteriye servis ediyoruz, herkesin verisi izole kalıyor" demek. Tenancy bunu sağlayan mekanizma.

Üç model var, hibrit çoğu zaman doğru cevap

  • Single-tenant: Her müşteri kendi DB'si, kendi sunucusu. Yönetim cehennemi, ama kurumsal müşteri "verim sizinkilerle aynı yerde olmasın" diyorsa böyle yapılır.
  • Multi-tenant shared schema: Tek DB, tüm tablolarda tenant_id kolonu. En yaygın. Junior'ın varsayılan başlangıç noktası olmalı.
  • Multi-tenant separate schema: Tek DB, her tenant kendi Postgres schema'sı. Backup/restore kolay, izolasyon güçlü, ama 10K+ tenant'ta migration zorlaşır.
  • Hybrid (Shopify pattern): Küçük müşteriler shared schema'lı "pod"larda, dev müşteriler dedicated pod'da. shop_id ile pod routing.

Üç framework, üç farklı yorum

  • Frappe: site-per-bench. Her tenant ayrı virtual host. site1.local, site2.local. Filesystem'de ayrı dizin.
  • Salesforce: tek dev kernel, her tabloda OrgID. 200K+ tenant aynı tablolarda; OrgID query layer'da otomatik filter. Bu en zoru ama en çok ölçeklenen.
  • Shopify: her merchant store_id, MySQL'de pod sharding. Black Friday'de saniyede 80K request'i bu yapıyla kaldırdı.
Gerçek dünya örnekleri
  • Notion: senin workspace'in var (Personal), şirketin workspace'i (Acme Inc) var. Aynı kullanıcı ikisinde de üye, workspace switcher'la geçiş yapar. workspace_id = tenant_id.
  • Slack: workspace = tenant. Her message, channel, file bir workspace'e ait. Bir workspace'in mesajı diğerinde aratılmaz, görünmez.
  • Shopify (#12 senaryon): her merchant kendi store'una sahip — kendi customer DB'si, kendi ürünleri, kendi domain'i, kendi tema'sı. A store'un customer'ı B store'una sızmamalı, KVKK / GDPR ihlali olur.
  • Linear / Jira: organization = tenant. Issue'lar, project'ler hep organization'a bağlı. Aynı kullanıcı 3 organization'da üye olabilir.
  • HRMS (#9): tenant = company. Türk şirketinin İK verisi Alman şirketininkiyle aynı tabloda ama tenant_id ile izole.
  • Muhasebe SaaS (#10): tenant = mali müşavir bürosu (her bürosunun ayrı mükellef portföyü var — burada bir "tenant of a tenant" sorunu çıkar, açıklaması aşağıda).
  • LMS (#6): tenant = okul/kurum. Aynı LMS kodu 500 okulu sunar. Bir okulun öğrencisi diğerinin sınıfını görmez.

Niye CORE

Day-1'den itibaren her satırda tenant_id olmalı, her query otomatik WHERE tenant_id = ? filtrelemeli (Authz ile birlikte). Sonradan retrofit'lemek = tüm tabloları rebuild + tüm query'leri rewrite + 6 ay kayıp + güvensiz geçiş penceresi.

Junior tuzağı #1 — "Önce tek müşteri için yapayım"

"İlk müşterim için tutarlı çalışsın, sonradan çoklarım" düşüncesi. Kaybedilen 6 ay. Çünkü her sorguya tenant filtresi eklemek mekanik iş değil — bazı sorgular cross-tenant olmalı (admin dashboard), bazıları kesin olmamalı (auth flow). Doğru ayrımı baştan yapmak retrofit'ten 10x ucuz.

Junior tuzağı #2 — Tenant ≠ User

Bir user birden çok tenant'ta üye olabilir (Slack workspace switcher gibi). User ↔ Tenant many-to-many. Tek bir user.tenant_id alanı yetmez. Membership tablosu lazım (membership: user_id, tenant_id, role, joined_at).

Junior tuzağı #3 — Tenant ≠ Site (CMS'te)

Bir tenant birden çok site (subdomain, custom domain) işletebilir. CMS turunda konuşulmuştu: bir kurumsal müşterin 5 marka sitesi olabilir, hepsi aynı tenant'a ait. site tablosu ayrı, tenant_id'ye bağlı.

Junior tuzağı #4 — Tenant'ın tenant'ı (muhasebe senaryon)

Mali müşavir bürosu senin müşterin. Bürosunun 50 mükellefi var, her mükellefin defteri ayrı tutulmalı. Burada iki seviyeli tenancy var: tenant (büro) + sub-tenant (mükellef). Bunu modellemezsen büro tüm mükelleflerin defterini karıştırır. Çözüm: tenant_id (büro) + company_id (mükellef şirket) ikilisi her satırda.

Layer 0 — Kernel

Event Bus + Polyglot Runtime Bridge

Junior'a: Module'ler birbirine doğrudan fonksiyon çağırmaz. Event publish ederler, ilgilenenler subscribe eder. Loose coupling'in motoru.

Klasik anti-pattern (junior çoğunlukla böyle başlar)

# Order module
def place_order(order):
    save(order)
    inventory.decrement_stock(order.items)      # tight coupling
    notification.send_email(order.customer)     # tight coupling
    loyalty.add_points(order.customer, order.total)  # tight coupling
    analytics.track("order_placed", order)      # tight coupling

Yarın 5. module geldiğinde Order module'üne 5. satır eklersin. Order module artık 5 module'ün varlığını bilmek zorunda. Modülerlik yok.

Doğrusu — event publish

# Order module
def place_order(order):
    save(order)
    bus.publish(OrderPlaced(order_id=order.id, items=order.items, customer=order.customer))
    # Order module bu kadarını bilir. Kim dinliyor, umurunda değil.

# Inventory module (ayrı dosya, ayrı geliştirici, ayrı deploy)
@bus.on(OrderPlaced)
def decrement_stock(event):
    for item in event.items:
        stock.decrement(item.sku, item.qty)

# Notification, Loyalty, Analytics module'leri de aynı şekilde subscribe eder

Aynı kod, üç ölçek (sihir bu)

Event bus'ın interface'i tek, implementation'ı deployment profile'a göre değişir:

  • Shared hosting (5 €/ay VPS): in-process Python event handler. Async loop. Tek process.
  • Orta ölçek (10K req/s): Redis Streams. Birden çok worker, persistence var.
  • Büyük ölçek (Twitter benzeri): Kafka. Saniyede milyon event, partition'lı, replay'lenebilir.

Module kodu hiç değişmez. Sadece config.event_bus_backend = "redis" ya da "kafka". Spring Modulith bunu yapar. Sen de yapmalısın.

Polyglot Runtime Bridge — Python ve TypeScript birinci sınıf

Sen istiyorsun ki geliştirici hook'u Python veya TypeScript ile yazsın. Üç runtime strateji var:

Embedded JS (shared hosting)

PythonMonkey (SpiderMonkey) ya da STPyV8 (V8) — Python process'in içinde JS engine. Network yok, latency düşük. 5 €/ay VPS'te single-process çalışan kullanıcılar için.

Sidecar (production default)

Node/Deno ayrı proses, gRPC ile Python core'a bağlı. Dapr modeli. Production'da güvenilir, izolasyon var, bağımsız ölçeklenir.

WASM / Extism (untrusted plugin)

Marketplace plugin'leri — başkasının kodu. Sandbox'lı, capability-based. "Network erişimi yok, disk yok, sadece bu fonksiyon" diye kısıtlanabilir.

Aynı interface

Üç runtime de aynı hook contract'ı implement eder. Geliştirici hangisini kullandığını bilmez, framework deployment profile'a göre seçer.

Geliştirici tarafı aynı görünür:

// hooks/order.ts — TypeScript
export const onBeforeSave = (doc: Order) => {
  if (doc.total < 0) throw new Error("Total negatif olamaz");
  doc.totalWithTax = doc.total * 1.20;
};
# hooks/order.py — Python
def on_before_save(doc):
    if doc.total < 0:
        raise ValueError("Total negatif olamaz")
    doc.total_with_tax = doc.total * 1.20

Kritik DDD kuralı — aggregate boundary crossing yasak

TS hook'unun Python aggregate'i üzerinde açık transaction tutmasına izin verme. Dağıtık transaction (2PC) çağı geçti, hem yavaş hem fragile. Doğru pattern'ler:

  • Synchronous hook = pure function. Python core transaction açar, TS hook'a aggregate kopyasını verir, TS hook decision/mutation döner, Python core commit eder. Frappe'nin doc_events contract'ı tam bu.
  • Cross-aggregate effect = outbox + saga. TS hook farklı aggregate'i değiştirmek istiyorsa, transaction'da olmaz. Outbox'a event yazılır, ayrı consumer asenkron işler, başarısızlıkta compensating action.
Gerçek dünya örnekleri
  • Slack: bir mesaj postlandığında Slack içinde event firee eder. Bot'lar (Linear, GitHub, Datadog) subscribe eder. Slack core'da "Linear" diye bir kavram yok — Linear bot Slack event API'sini dinler.
  • Stripe webhook: Stripe internal event bus'ını dışarıya açar. payment_intent.succeeded event'i webhook olarak senin endpoint'ine düşer. Senin sistemine event publish'i evrensel olarak böyle yapılır.
  • AWS S3 → Lambda: S3'e dosya konunca ObjectCreated event'i otomatik Lambda'yı tetikler. Bulutta event bus pattern'inin canonical örneği.
  • E-ticaret (#3) senaryo: OrderPlaced → Stock decrement + Email + Loyalty + Recommendation update + Tax record + Analytics event. Hepsi paralel, hiçbiri Order module'ü bilmez.
  • Sosyal medya (#14): PostCreated → Fan-out + Search index + Moderation queue + Trend update. Fan-out'ın yapacağı iş 100M'lik bir push olabilir; ana transaction içinde olamaz, async.

Niye CORE

Module'ler birbirinden habersiz yaşayacaksa event bus şart. Sonradan eklersen tüm module'ler birbirine kavradığı için (tight coupling) module sistemi çöker. Polyglot bridge core'da çünkü hook contract aynı olmadan TS modülü TS modülünden farklı doğmaz.

02Scale Primitives (kernel-adjacent)

Bunlar Layer 0 değil, ama kernel'in save pipeline'ına örülmesi gerek. Sonradan retrofit imkansıza yakın. "Modül yazarım sonra" diyemezsin çünkü her DocType save'i bu pipeline'dan geçer.

Scale primitive

Transactional Outbox

Junior'a: "Order'ı DB'ye yazıp Kafka'ya event publish et" yazınca şu sorun: ikisi iki ayrı transaction. Biri başarılı, biri başarısız olabilir. Dual-write problem.

Naif yaklaşım (yanlış):

def place_order(order):
    db.save(order)            # ✓ commit
    kafka.publish(event)      # ✗ network down → event kayıp, Order var
    # ya da
    kafka.publish(event)      # ✓ yayınlandı
    db.save(order)            # ✗ DB crash → event var, Order yok

Outbox pattern:

def place_order(order):
    with db.transaction():
        db.save(order)
        db.save(OutboxEntry(event_type="OrderPlaced", payload={...}, status="pending"))
    # ✓ ikisi de aynı transaction'da, atomic

# Ayrı bir worker (relay):
def outbox_relay():
    while True:
        for entry in db.fetch_pending_outbox(limit=100):
            kafka.publish(entry.payload)
            entry.mark_as_published()

Process crash olursa: Order DB'de var, outbox row'u DB'de var (committed). Worker tekrar başlar, kaldığı yerden devam eder. Event garantili yayınlanır.

Niye core'a örülmeli: her DocType save'i save pipeline'dan geçer. Outbox kodunu module'lere bırakırsan unutan olur. Pipeline'da yapınca her aggregate write otomatik outbox-aware.

Retention

Outbox tablosu büyür. Bir fintech 2 milyar satıra ulaşmış. Retention politikası gün-1: published row'ları 7 gün sonra arşivle veya başka tabloya taşı.

Scale primitive

CQRS Projections (Materialized Read Models)

Junior'a: Yazma ve okuma model'ini ayır. Yazma model'i transaction-safe, normalize. Okuma model'i hız-için-tasarlanmış, denormalize, eventual consistent.

Klasik problem: E-ticaret dashboard'unda "müşterinin son 30 günde toplam harcaması". Naif:

SELECT SUM(total) FROM orders WHERE customer_id = ? AND created_at > NOW() - INTERVAL '30 days'

10M sipariş, 1M müşteri olunca her dashboard request'inde aynı query. Yavaş. Cache'liyorsun, stale.

Projection ile:

# Bir tabloya CustomerStats projection'ı kur
# customer_id, total_spent_30d, order_count, last_order_at

@bus.on(OrderPlaced)
def update_customer_stats(event):
    stats = db.get_or_create_stats(event.customer_id)
    stats.total_spent_30d += event.total
    stats.order_count += 1
    stats.last_order_at = event.timestamp
    db.save(stats)

Dashboard request'inde tek SELECT, hazır agrege. Eventual consistency = event işlenene kadar 1-2 saniye gecikme olabilir. Çoğu zaman kabul.

Gerçek dünya
  • Twitter timeline: PostCreated event'inden materialize edilmiş per-user timeline. Okuma anında JOIN yok, denormalize hazır.
  • YouTube view count: yazma yapana view++, okuma yapana cached projection.
  • Stripe dashboard: "this month revenue" — her ödeme event'i projection'ı update eder.
  • Sosyal medya feed: bütün fan-out işi projection.

Niye core'da: "Bu DocType'ın şu projection'ı var, şu event'lere subscribe oluyor" deklaratif metadata olmalı. Module'lere bırakmazsın, framework'ün otomatik rebuild edebilmesi lazım (replay).

Scale primitive

Hot Counter

Junior'a: Like sayacı, view sayacı, follower sayacı — milyonlarca eşzamanlı kullanıcı aynı satıra yazıyor. Klasik UPDATE post SET likes = likes + 1 WHERE id = ? SQL'inde lock contention, throughput çöker.

Strateji:

  • Shared hosting: normal DB column. Yeterli.
  • Orta ölçek: Redis INCR. Periodic flush DB'ye (her 10 saniyede bir).
  • Büyük ölçek: sharded counter — like_counter_post_123_shard_0, shard_1, ... toplama anında SUM. Approximate counter, kesinlik feda, hız kazanılır.

Module API'si tek:

counter.increment("post:123:likes")
counter.read("post:123:likes")  # approximate ok

Backend strateji deployment profile'a göre. Module geliştirici hangi strateji aktif bilmez.

Niye core'da: Stok turundaki "miktar = hareketlerin toplamı" disiplini muhasebe için doğrudur (ACID). Sosyal medya counter'ı için yanlış pattern — okuma amplifikasyonu, eventual consistency tolere edilir. İki ekosistem aynı framework'te yaşayacaksa counter abstraction core'da olmalı.

Scale primitive

Background Workers + Scheduler

Async job queue: Frappe RQ, Celery, BullMQ benzeri. Outbox relay, projection rebuild, fan-out worker, cron job hepsi burada.

  • Job tipleri: immediate (queue'ya at, sıradaki worker alır), scheduled (15 dakika sonra çalışsın), recurring (her gün saat 02:00).
  • Priority: kritik job'lar (ödeme webhook) yüksek; bilgilendirme mail'i düşük.
  • Retry: exponential backoff, dead letter queue.
  • Idempotency: aynı job iki kez çalışsa bile sonuç aynı (network retry'a karşı koruma).

Shared hosting: in-process thread pool veya APScheduler. Production: ayrı worker pod'ları, Redis/RabbitMQ queue. Aynı API.

03Layer 1 — In-tree core (default gelir, swappable)

Bu domain'ler Frappe core'da var, çünkü 15 üründen 13'ünde geçiyor. Ama Medusa "Infrastructure Module" disiplini ile: interface arkasında, topluluk değiştirebilir.

Layer 1 — In-tree

Party / Contact

Junior'a: "Customer" yazma. "Party" yaz. Çünkü Customer, Supplier, Employee, Lead, Patient, Student — hepsi farklı role oynayan aynı kişi olabilir. Generic abstraction Party, business semantic role module katmanında.

Frappe'nin verdiği ders

Frappe core'da Contact ve Address DocType'ları var. ERPNext'te Party Type (Customer / Supplier) DocType'ı var. Yani generic kişi/adres core'da, business role module'de. Sen aynı çizgiyi çiz.

Yapı

Person (kişi)
  - first_name, last_name, birthdate, gender, ...
Organization (şirket)
  - legal_name, tax_id, founded_at, industry, ...
Party = Person | Organization (polymorphic)
ContactPoint
  - email, phone, website (Party'ye N-to-1)
Address
  - country, city, district, street, postal_code (Party'ye N-to-1)
PartyRole (module katmanında)
  - customer, supplier, employee, lead, patient, student, ...
Gerçek dünya
  • Real estate (#1): Party = aynı kişi hem alıcı (buyer role), hem satıcı (seller role), hem danışman (agent role) olabilir. Üç ayrı tablo değil, tek Party + üç role.
  • Hastane: Party = doktor, hemşire, hasta, refakatçi. Doktor aynı zamanda hasta olabilir (kendi muayenesi). Tek Party.
  • Üniversite: öğrenci mezun olunca aynı Party, role değişir (alumni). LinkedIn benzeri network kurabilirsin.
  • E-ticaret B2B (#2): bir şirket hem alıcı hem satıcı olabilir (distributor pattern). Tek Organization, dual role.
  • CRM: lead → contact → customer dönüşümü. Aynı Party, role evrimi. Yeni tablo değil, role değişimi.
Tuzak — "Customer" core'a koyma

Sosyal medyada (#14) "Customer" yok. CMS'te (#5) yok. LMS'te (#6) yok. Eğer Customer'ı core'a koyarsan bu 3 ürün için ölü kavram taşımış olursun. Generic kısım (Person/Org/Contact/Address) core, "Customer" rolü module'de.

Layer 1 — In-tree

File & Storage

Dosya upload, thumbnail, version, metadata, signed URL, CDN, virus scan, MIME validation.

Driver pattern — backend swappable

storage.put(file, key)     # interface
storage.get(key)
storage.signed_url(key, expires_in=3600)

# Implementation pluggable:
# - LocalDiskDriver (shared hosting, /var/storage)
# - S3Driver (AWS, default production)
# - R2Driver (Cloudflare R2, S3-uyumlu, egress free)
# - AzureBlobDriver
# - GCSDriver

Module değiştirebilmeli. Bir kullanıcı self-hosted MinIO kullanmak isteyebilir — yeni driver yazar, plug eder.

Örnekler
  • Profil fotoğrafı: upload → automatic resize (avatar 64, profile 256, hero 1024) → CDN URL döner.
  • PDF rapor: server-side oluşturuldu → S3'e konuldu → signed URL ile 5 dakika geçerli download link.
  • Video (sosyal medya #14): upload → transcode pipeline (480p, 720p, 1080p, HLS) → thumbnail çıkar → moderation queue.
  • Drive ürünü (#7): Layer 1 file motoru yetmez, üstüne Block Storage (CAS — content-addressable storage, dedup, delta sync) modülü gerekir. Bu Layer 2/3.
  • Mali belge (#10): uzun süre saklama (7+ yıl, KVK / VUK), immutable, hash-verified.
Layer 1 — In-tree

Workflow & Process

Junior'a: State machine. Bir DocType'ın izlediği durumlar ve durumlar arası geçiş kuralları.

Workflow: PurchaseOrder
  states: [draft, pending_approval, approved, sent, received, closed]
  transitions:
    - from: draft → pending_approval, by: requester
    - from: pending_approval → approved, by: manager, condition: amount < 10000
    - from: pending_approval → approved, by: cfo, condition: amount >= 10000
    - from: approved → sent, by: purchasing_team
    - from: sent → received, by: warehouse
    - from: received → closed, by: accounting

İçerir:

  • State machine tanımı (deklaratif)
  • Transition rule + permission (kim hangi geçişi yapabilir)
  • Approval chain (manager + finance birlikte)
  • Timeout / escalation (3 gün onay yoksa CFO'ya eskale)
  • Branching (tutara göre farklı approver)
  • Audit trail (her geçiş kim/ne zaman)
Örnekler
  • HR izin onayı (#9): Çalışan talep → direct manager onay → HR onay → kabul. 14 gün üstüyse genel müdür eklenir.
  • E-fatura akışı (#10): Fatura draft → finance review → manager approve → GİB'e gönder → onaylı.
  • PR review (#11): draft PR → CI green → review request → 2 approval → merge.
  • İlan moderasyonu (#1, #14): kullanıcı submit → AI ön-tarama → eğer flagged: insan moderatöre → onay/red.
  • Refund akışı: müşteri talep → AI auto-approve (eğer kurallar tutuyorsa) → veya insan inceleme → tahsilat iadesi.

Niye core (Layer 1): 13 üründe geçiyor. Frappe core'da var. Drupal'da contrib. Sen core'da tut çünkü tekrar tekrar ihtiyaç duyulacak.

Layer 1 — In-tree

Notification

Channel-agnostic abstraction. Module geliştirici sadece notify(user, template, data) yazar. Framework kanal seçer (kullanıcının tercihine göre).

Kanallar:

  • Email (transactional + marketing ayrımı, SMTP/SES/SendGrid driver)
  • SMS (Twilio, NetGSM, Turkcell SMS — driver)
  • Push notification (iOS APNs, Android FCM, web push)
  • In-app notification (bell icon, read/unread state)
  • Slack / Teams / Discord webhook
  • Generic webhook (B2B müşteri kendi sistemine alır)

İçerir:

  • Template engine — Jinja/Handlebars, i18n-aware (kullanıcının diline göre template seç)
  • User preference — kullanıcı "marketing email istemiyorum, transactional gelsin" diyebilmeli
  • Throttling — saatte 1'den fazla aynı bildirimi atma
  • Batching / aggregation — "12 kişi gönderini beğendi" tek bildirim (sosyal medya pattern)
  • Delivery tracking — açıldı mı, click'lendi mi, bounce mu
  • Unsubscribe management — yasal zorunluluk, link kullanıcıyı pref sayfasına götürür
  • Queue + retry — SMTP down, sonra dene
Anti-pattern — her module kendi SMTP'sini kursun

Junior her modüle smtplib.SMTP("smtp.gmail.com", 587) yazarsa: config dağınık, throttling yok, queue yok, retry yok, unsubscribe link inconsistent. Notification core'a koymak = tek pipeline'dan geçer = compliance kolay.

Layer 1 — In-tree

Audit & Activity

Her değişikliğin kim/ne/ne zaman log'u. Append-only, immutable.

İki tür var, ayrı tutulmalı:

  • Audit log (compliance-yüzlü): "karaca@yandex.com kullanıcısı 2026-06-04 14:32:08'de Order#12345'in status alanını pending'den cancelled'a değiştirdi". Immutable, retention politikası ile (yasal süre kadar saklanır). Muhasebede (#10), HRMS'te (#9) zorunlu.
  • Activity feed (kullanıcı-yüzlü): "Ahmet bu issue'ya yorum yaptı (5 dk önce)". Soft delete OK, UX odaklı, expire olabilir.
Örnekler
  • Muhasebe (#10): Vergi denetiminde "12 ay önce şu hesap kalemini kim ne zaman değiştirdi" sorusu. Audit log olmasa hapis.
  • HRMS (#9): "Çalışanın maaşı 2025-03 → 2025-04'te kim onayladı". HR compliance.
  • GitHub PR: "Bu satırı 3 ay önce şu kişi şu commit'te ekledi". Activity feed.
  • Drive (#7): "Bu dosyayı bu hafta kim açtı, kim indirdi, kim değiştirdi". Hem audit hem activity.
  • SaaS support: "Müşteri X'in hesabına dün admin Y giriş yaptı (impersonation)". Audit.
Layer 1 — In-tree

Search & Indexing

Interface: search(query, filters, sort, pagination). Backend swappable.

  • Shared hosting default: SQLite FTS5 (Postgres tsvector). Yeterli. Sıfır ekstra altyapı.
  • Orta ölçek: Meilisearch, Typesense — typo-tolerant, fast, kolay setup.
  • Büyük ölçek: OpenSearch / Elasticsearch — milyarlarca doküman, complex faceting.
  • AI-first ek: pgvector / Pinecone / Qdrant — semantic search (embedding-based). "Beylikdüzü'nde deniz gören 3+1, 6M altı" doğal dil aramaya bu olmadan cevap veremezsin.

Event-driven indexing

DocType save'i → DocumentSaved event → search indexer worker → backend'e push. Projection pattern'in özelleşmesi. Module geliştiricinin "index güncelle" düşünmesi gerekmez.

Tuzak — Elasticsearch'ü kernel'e fuse etme

"Search core'da olsun" diye Elasticsearch'ü dependency yapma. Layer 1 = interface core'da, default SQLite, prod Elasticsearch swap. Frappe core'da search var (basit), ciddi search isteyen Meilisearch eklentisi takıyor. Medusa search'ü Infrastructure Module yapmış. Aynı çizgi.

Layer 1 — In-tree

Calendar

Event, recurring event (RRULE iCal standardı), timezone handling, availability, conflict detection.

İçerir:

  • Event entity (start, end, all-day, location, attendees)
  • Recurrence rule (RRULE) — "her pazartesi", "her ayın ilk perşembesi"
  • Exception (recurring serisinin bir oturumu iptal)
  • Timezone-aware (DST geçişleri, IANA zone)
  • iCal export (.ics dosyası)
  • CalDAV protocol (Apple Calendar, Outlook sync)
  • Availability query (boş slot ara)
Kullanım yerleri
  • HRMS izin takvimi (#9): Çalışanın izinli olduğu günler. Manager dashboard'da takım kapasitesi görür.
  • PMS deadline (#11): Sprint son tarihi, milestone, gate.
  • LMS canlı ders (#6): Webinar, sınıf, sınav. Zoom entegrasyonu, takvim invite.
  • CRM görüşme: Sales call, demo, follow-up. Calendly entegrasyonu.
  • Real estate (#1): Mülk görme randevusu, alıcı + satıcı + danışman üçlü slot.
  • Healthcare: Doktor randevusu, doctor availability + patient booking.

04Core sanılan ama core olmayan

Junior'ın "ben bunu core'a koyarım, çünkü her ürünün lazım olur" diyeceği ama yanılacağı kavramlar.

Billing core değil

Her ürünün billing modeli farklı: B2B sözleşme + vade (Alibaba #2), retail kupon + coin (AliExpress #3), escrow + milestone (#2), seat-based subscription (#12), usage-based (API ürünleri), donation (non-profit). Billing core'a fuse edersen tek modeli herkese dayatırsın. Layer 2 first-party module yap, ihtiyacı olan kurar. Frappe, WordPress, Odoo, Salesforce hepsi aynı çizgide.

Catalog / Pricing core değil

E-ticaret ürünleri için Layer 2 module. CMS, LMS, Drive, sosyal medya gibi ürünlerin Catalog'la işi yok. Core'a koymak ölü kavram taşımak.

Reporting / Analytics core değil

Frappe ve Salesforce core'a koymuş ama bu istisna. Sen Layer 2 yap — kullanan kurar. Genel raporlama (her DocType için liste, filter, export) zaten DocType engine'in özelliği. Cross-domain analytics (BI dashboard) module.

AI / Agent core değil

"AI-first framework" dediğine bakma, AI domain kendi başına Layer 2 module. Core'da olan şey: AI'ın bağlandığı substrate (event bus, schema, embeddings için pgvector). AI agent davranışları, prompt template'leri, model routing — module katmanında. Çünkü farklı ürünlerin AI ihtiyacı tamamen farklı (sosyal moderation ≠ muhasebe categorization ≠ stok forecasting).

Logging / Monitoring / Caching

Bunlar infrastructure concern, "domain" değil. Sen structlog kullanırsın, OpenTelemetry kurarsın, Redis cache adapter Layer 1 olur. Bunları "domain" olarak kavramlaştırmak DDD'yi su sıkar.

Feed / Timeline — sosyal medya hayranı için

Bu kritik. Sosyal medya turunda konuşulmuştu. "Fan-out core'da olsun" diye düşünebilirsin çünkü "scale primitive". Yanıt: primitive'ler core'da, policy module'de. Event bus, outbox, async worker, hot counter, projection — core'da. Push vs pull seçimi, celebrity threshold (~10K follower), hybrid merge — Layer 2 Feed module. Çünkü 13 ürünün feed'i yok; core'a fuse edersen onları gereksiz overhead taşır.
